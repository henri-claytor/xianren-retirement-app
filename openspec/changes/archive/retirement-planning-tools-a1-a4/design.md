## Context

嫺人退休理財課程目前以 Google Sheet 提供學員試算，缺乏視覺化、個人化與持續追蹤能力。本次開發退休前規劃模組（A1~A4），為原生 APP（iOS + Android）的核心計算工具層，串接公司既有財經資料 API，並透過 Page 系統與內容（B）、討論（C）模組組裝。

**技術限制：**
- 使用公司既有用戶系統（SSO），不另建帳號系統
- 財經資料（台股即時、美股即時、基金T-1淨值、台銀匯率、ETF/基金 bond_ratio）均透過公司 API 取得
- 蒙地卡羅模擬（10,000次）需後端處理，預估 1-3 秒
- bond_ratio 月頻率更新，來源為上手資料庫

## Goals / Non-Goals

**Goals:**
- 實作 S1 財務現況輸入（含台股/美股/基金標的搜尋與即時報價）
- 實作 S2 三桶金現況總覽（自動歸桶 + 手動覆蓋 + 歸桶變動通知）
- 實作 S3 通膨侵蝕模擬器（多情境視覺化曲線）
- 實作 A1 退休目標設定（通膨調整 + 4%/3%提領法則 + 年金終值）
- 實作 A2 壓力測試（蒙地卡羅 10,000次 + 多壓力情境 + 系統建議）
- 實作 A3 資產配置建議（三桶金再平衡 + 標的建議清單）
- 實作 A4 定期追蹤更新（財務快照 + 歷史趨勢 + 推播通知）
- 所有計算結果透過 API 存入公司內部 DB，供用戶行為分析

**Non-Goals:**
- 退休後提領模組（B1~B4）不在本次範圍
- 整合儀表板（D1）不在本次範圍
- 內容模組（B）與討論模組（C）整合不在本次範圍
- 免費版功能限制邏輯（另行規劃）
- 財務顧問諮詢或投資建議（僅提供計算與參考方向）

## Decisions

### 決策 1：蒙地卡羅模擬在後端執行

**選擇**：後端非同步計算，前端顯示 Loading 狀態，完成後推送結果。

**理由**：10,000 次模擬 × 50 年時間序列 = 500,000 次計算，在行動裝置前端執行會有效能與電量問題。後端可快取相同參數的計算結果，避免重複運算。

**替代方案**：前端 WebWorker 執行 → 跨裝置效能差異大，iOS 限制多，捨棄。

**快取策略**：以 `user_id + 輸入參數 hash` 為快取 key，TTL 24 小時。參數變動時清除快取重新計算。

---

### 決策 2：三桶金歸桶規則

**選擇**：
- 現金/定存 → 短期桶
- 儲蓄險 → 中期桶
- 股票（所有個股）→ 長期桶
- ETF/基金：bond_ratio ≥ 55% → 中期桶；< 55% → 長期桶
- 不動產自住 → 獨立呈現；出租 → 長期桶

**理由**：符合嫺人課程三桶金理念（短期流動、中期固收、長期成長），個股風險較高歸長期符合分散投資原則。

**邊界條件**：
- bond_ratio 剛好 55% → 歸中期桶（含等於）
- bond_ratio 資料缺失 → 暫歸長期桶 + 前端警示
- 使用者可手動覆蓋，覆蓋紀錄另存，供變動比較

**bond_ratio 觸發重新確認條件**：本次更新值與手動覆蓋當下值差異 > 5% 時，推播通知請用戶確認。

---

### 決策 3：財務快照設計（A4）

**選擇**：每次更新建立新快照（append-only），保留完整歷史，不覆蓋舊資料。

**理由**：支援歷史趨勢圖（成功率、資產、三桶金配置變化），並提供用戶行為分析資料。`is_active` 欄位標記最新快照。

**替代方案**：每次更新覆寫同一筆資料 → 無法呈現趨勢，捨棄。

---

### 決策 4：有價證券自動更新 + 手動輸入並行

**選擇**：
- 有價證券（台股/美股/基金）：透過 API 自動抓取即時/T-1 市值
- 現金、不動產、儲蓄險、負債：用戶手動輸入

**理由**：有價證券有明確報價來源，自動化可提升準確性與使用者體驗。現金與不動產無外部報價，僅能手動輸入。

---

### 決策 5：標的建議池設計

**選擇**：系統維護一份推薦標的池（中期桶：債券型ETF/基金；長期桶：股票型ETF），依費率低→規模大排序，優先顯示嫺人課程提及標的。

**篩選條件**：
- 中期桶：bond_ratio ≥ 55%、費率 < 0.5%
- 長期桶：bond_ratio < 55%、費率 < 1%

**不主動建議個股加碼**：已持有個股標注「已持有」，但不主動建議買進，符合嫺人「不押寶個股」原則。

---

### 決策 6：系統建議邏輯（A2）

當退休成功率 < 85% 時，系統自動試算以下方案並排序：
1. 提高每月儲蓄（+5,000 / +10,000 / +20,000）
2. 調整長期桶比例（60% / 65% / 70%）
3. 延後退休年齡（+1 / +2 / +3 年）
4. 降低退休支出（-10% / -20%）

輸出最多 3 個成功率提升幅度最大的方案。

## Risks / Trade-offs

**[風險] 蒙地卡羅模擬超時（>3秒）** → 實作進度條 + 背景計算，完成後推播通知。初次進入設定頁預先觸發計算。

**[風險] bond_ratio 資料月更新，可能有標的資料缺失** → 前端顯示警示「資料待更新，暫歸長期桶」，不影響整體計算。

**[風險] 美股即時報價時區問題（台灣非交易時間）** → 顯示最後成交價 + 更新時間，明確標注「盤後價格」。

**[風險] 通膨/報酬率假設與實際差異大** → 提供情境比較（樂觀/標準/保守），並在 UI 明確標注「此為試算參考，非投資建議」。

**[Trade-off] 快照 append-only 導致 DB 資料量增長** → 每位用戶預估每年 2-4 次快照，資料量可控。定期清理超過 10 年的歷史快照。

**[Trade-off] 標的建議池需人工維護** → 初版手動維護，後續可依交易量/費率自動更新。

## Migration Plan

1. 後端先行：建立 5 張 DB 資料表、7 支 API
2. 財經資料串接：確認台股/美股/基金/匯率/bond_ratio API 格式與授權
3. 計算引擎實作：通膨模擬 → 退休目標 → 蒙地卡羅 → 配置建議（依序驗證）
4. 前端元件開發：S1→S2→S3→A1→A2→A3→A4→共用元件整合
5. Page 系統整合：將 A 模組元件接入 Page 系統，供 B/C 組裝
6. 測試：計算結果與原 Google Sheet 交叉驗證，誤差 < 1%

**Rollback**：各 API 獨立部署，可單獨回滾。前端元件版本化，可指定版本回退。
