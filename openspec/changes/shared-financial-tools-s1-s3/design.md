## Context

S1~S3 是整個退休規劃 APP 的數據基礎層，所有計算工具（A1~A4、B1~B4）均依賴此層的輸入資料。本層需優先開發並確保 API 介面穩定，避免上層模組頻繁調整。

**技術限制：**
- 財經資料（台股即時、美股即時、基金T-1、台銀匯率、bond_ratio）均透過公司既有 API 取得
- bond_ratio 月頻率更新，來源為上手資料庫廠商
- 原生 APP（iOS + Android）
- 用戶資料存入公司內部 DB，供用戶行為分析
- S1~S3 封裝為 Page 系統相容元件

## Goals / Non-Goals

**Goals:**
- 實作 S1 財務現況輸入（標的搜尋 + 即時報價 + 手動輸入 + 支出三分類 + 負債 + 即時財務摘要）
- 實作 S2 三桶金歸桶引擎（自動歸桶 + 手動覆蓋 + bond_ratio 異動通知 + 視覺化）
- 實作 S3 通膨侵蝕模擬器（多情境曲線 + 滑桿即時更新 + 購買力說明）
- 所有 S 層 API 回傳結構需穩定，作為 A/B 層的合約

**Non-Goals:**
- A1~A4 退休前規劃計算（另立 change）
- B1~B4 退休後提領計算（另立 change）
- D1 整合儀表板（另立 change）
- 免費版功能限制邏輯

## Decisions

### 決策 1：S1 有價證券採標的級輸入（非總值輸入）

**選擇**：用戶逐筆輸入標的代號 + 股數/單位數，系統自動抓報價計算市值。

**理由**：作為財經數據公司，可串接即時報價，提供比 Google Sheet 更精準、更省力的資產盤點體驗。標的級輸入也能支援 S2 的自動歸桶（需知道各標的的 bond_ratio）。

**替代方案**：僅輸入各類別總值（如舊版 Google Sheet）→ 無法支援自動歸桶與損益計算，捨棄。

---

### 決策 2：美股非交易時段顯示最後成交價

**選擇**：美股非交易時段時，顯示最後成交美元價 + 台銀匯率換算台幣，並標注「盤後價格」及更新時間。

**理由**：台灣用戶主要操作時間為台灣白天，與美股交易時段不重疊。顯示最後成交價並清楚標注，比顯示空值更有用。

---

### 決策 3：支出採嫺人三分類法

**選擇**：生活必需 / 生活風格 / 階段性支出三類，其中生活風格佔比建議 ≤ 30%，階段性支出需設定結束年齡。

**理由**：這是嫺人課程的核心方法論，APP 需完整呈現此框架，讓學員在實際使用時強化課程觀念。

**生活風格佔比計算**：生活風格小計 ÷ 月支出合計（三類總和）× 100%。

---

### 決策 4：三桶金歸桶為後端計算，前端直接顯示結果

**選擇**：歸桶邏輯（assignBucket）在後端執行，前端僅呈現結果，不在前端重複計算。

**理由**：bond_ratio 資料量大，集中在後端可統一維護歸桶邏輯，避免前後端邏輯不同步。手動覆蓋也由後端管理，前端僅發送覆蓋請求。

---

### 決策 5：S3 通膨模擬為純前端計算

**選擇**：S3 的通膨曲線計算在前端執行，不需後端 API（僅最終結果儲存時呼叫 API）。

**理由**：通膨模擬參數少、計算簡單（複利公式），前端即時計算可實現滑桿拖動即時更新曲線的體驗，若需後端往返會有明顯延遲。

**替代方案**：後端計算 → 每次滑桿調整都需 API call，延遲高，UX 差，捨棄。

---

### 決策 6：S1 輸入資料為 A/B/D 層的唯一數據來源

**選擇**：A1~A4、B1~B4、D1 所有計算均以 S1 最新快照的財務資料為輸入，不允許在各工具中各自維護財務資料。

**理由**：確保資料一致性，避免用戶在不同工具中看到不同的財務基礎。

**合約**：S1 的 financial_snapshots 表為所有上層工具的數據來源，上層工具以 snapshot_id 關聯。

---

### 決策 7：bond_ratio 缺失時暫歸長期桶並警示

**選擇**：bond_ratio 缺失 → 暫歸長期桶 + 前端顯示警示，不阻擋用戶繼續使用。

**理由**：阻擋用戶使用會造成糟糕體驗，暫歸長期桶（較保守）對計算結果影響可接受，並清楚告知用戶。

## Risks / Trade-offs

**[風險] 美股/基金報價延遲導致市值不準確** → 所有自動抓取的市值均標注資料時間戳，讓用戶知道是哪個時間點的價格。

**[風險] bond_ratio 月更新資料可能有缺漏** → 缺漏時暫歸長期桶並警示，月更新後自動修正，同時通知有手動覆蓋的用戶確認。

**[風險] 用戶輸入資料量大（多筆持倉）導致 S1 載入慢** → 標的搜尋採防抖動（debounce 300ms），報價批次請求而非逐筆請求。

**[Trade-off] 前端計算 S3 → 無法在伺服器端儲存每次模擬參數** → 用戶最終確認時才儲存結果，中間的滑桿調整不儲存，可接受。

**[Trade-off] 支出三分類增加輸入複雜度** → 透過分步驟引導（Step 1~5）降低認知負擔，並提供預設值（帶入上次填寫資料）。

## Migration Plan

1. 優先完成 S 層 API 介面定義（OpenAPI spec），讓 A/B 層開發可並行
2. 財經資料 API 串接驗證（台股 → 美股 → 基金 → 匯率）
3. bond_ratio 資料匯入與歸桶引擎驗證
4. S1 前端元件開發（最複雜，含標的搜尋 + 即時報價）
5. S2 前端元件開發（依賴 S1 資料 + 歸桶引擎）
6. S3 前端元件開發（純前端計算，可獨立開發）
7. Page 系統元件封裝

**Rollback**：S 層 API 獨立部署，版本化管理，回滾不影響 A/B 層（A/B 層使用已儲存的 snapshot 資料）。

## Open Questions

- 台股/美股標的搜尋的資料庫涵蓋範圍需與資料團隊確認（目前預期：台股含指數/個股/ETF、美股含指數/個股/ETF、台灣核備基金境內+境外）
- S1 輸入資料的自動儲存時機：即時儲存（每次修改）vs 用戶手動確認後儲存？建議後者，減少 DB 寫入頻率
- 生活風格佔比警示門檻（30%）是否為固定值，或可讓用戶自訂？
- S3 模擬結果是否需要存入 DB（目前設計為前端計算不強制儲存）？
